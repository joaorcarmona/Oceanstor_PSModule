### Native Pipeline & Context Architecture

- [ ] **Fix Multi-Object Pipeline Input (`Get-DMlun | Remove-DMLun` silently drops all but one item)**
  - **Confirmed real, not theoretical**: zero of 127 public commands have a `begin {}`/`process {}`/`end {}` structure. Verified empirically that an advanced function with a pipeline-bound parameter but no `process {}` block only *executes its body once*, using whichever pipeline object was bound *last* — piping 3 LUNs into a command like this runs it once, silently, on the 3rd. No error, no warning.
  - Today `WebSession` has `ValueFromPipeline`/`ValueFromPipelineByPropertyName` on **all 127** commands, but the actual identifying parameter that would make bulk pipelines useful (`LunName`, `HostName`, etc.) has it on only **8**. Fix needs both: add pipeline binding to the per-item identifying parameter for `Get-`/`Set-`/`Remove-`/`Rename-`/`Add-` families, *and* wrap each command body in `begin {}` (resolve `$session` once) / `process {}` (act on the current item) / `end {}` (only if a family needs a post-loop summary).
  - **Two behaviors the fix must preserve, both confirmed as intended usage (not hypothetical)**:
    1. "Connect and forget" — a single connection, then `Get-DMlun | Remove-DMLun` or `Get-DMlun | Add-DMLunToLunGroup` with no `-WebSession` ceremony anywhere.
    2. Concurrent multi-array sessions — `$storage1 = Connect-deviceManager -Hostname 10.0.0.1 -PassThru`, `$storage2 = Connect-deviceManager -Hostname 10.0.0.2 -PassThru`, then `Get-DMlun -WebSession $storage1` and `Get-DMlun -WebSession $storage2` are independently correct, and objects from each must keep hitting *their own* array when piped onward — not whichever session happens to be cached module-wide.
  - **Design decision #1 — RESOLVED, infrastructure already exists**: 38 of 39 typed return classes (`OceanstorLunv6`, `OceanstorLunv3`, `OceanStorHost`, etc. — every one except the session class itself) already carry `hidden [pscustomobject]$Session` / `$WebSession`, set at construction to whichever session fetched them. Their instance methods (`.Delete()`, `.Rename()`, `.Expand()`, `.NewSnapShot()`, `.GetSnapShots()` on the LUN/Host classes) already correctly use `$this.Session` — this is a working, tested mechanism for behavior 2 today, just via method calls instead of piped cmdlets. The gap is that the *cmdlets* don't consume this per-object session yet. Concretely, per command:
    - Drop `ValueFromPipeline` (by-value) from the `WebSession` parameter. This isn't just unused — it's live: a piped typed object (e.g. a LUN) coerces into a `[pscustomobject]`-typed parameter, so today `Get-DMlun | Remove-DMLun` actually tries to bind the whole LUN object into `-WebSession` by value, not into the identifying parameter. That's the actual mechanism behind the "silently drops all but one item" bug for by-value cases, not just the missing `process{}` block.
    - Keep/add `ValueFromPipelineByPropertyName` on `WebSession`, matching the object's `Session` (or `WebSession`) property — already present on every typed object, no new plumbing needed there.
    - Give the identifying parameter (`LunName`, `HostName`, etc.) `ValueFromPipelineByPropertyName = $true` **plus an `[Alias(...)]` matching the class's actual property name** — checked `Set-DMLun.ps1`: `LunName` already has `ValueFromPipelineByPropertyName = $true` today, but it silently never binds, because the LUN class's property is `Name`, not `LunName`, and there's no alias bridging them. The earlier "8/127 already have it" count is therefore optimistic — presence of the attribute was checked, not whether the property name actually lines up with the source class. Re-audit per command against the real class property name, not just grep for the attribute.
    - Per-item session resolution in `process {}`: explicit `-WebSession` argument (if the user passed one) > the piped object's own `Session` property (behavior 2) > `$script:CurrentOceanstorSession` (behavior 1, the default).
  - **Design decision #2 — still open**: bulk-pipeline error handling — if item 3 of 10 piped LUNs fails (e.g. a REST error via `Assert-DMApiSuccess`), should the whole pipeline stop (current single-item throw semantics, extended naturally), or should `process {}` catch per-item and continue, reporting failures at the end? Different modules do this differently (compare `Remove-Item -ErrorAction Continue` semantics); pick one and apply it consistently.
  - **Scope**: large, comparable to the REST-error-mapping refactor (68 files), but smaller than it first looked — the hard part (session travels with the object) is already built and battle-tested via the class layer; this is wiring cmdlet parameter binding + `begin`/`process`/`end` to use what's already there, not inventing new session-tracking machinery. Likely touches most `Get-`/`Set-`/`Remove-`/`Rename-`/`Add-` commands. Plan and scope properly (like that refactor) rather than editing files ad hoc; don't start with `Get-*` itself, since piping *into* a filter-returning `Get-` command is rarely useful — the real value is piping `Get-*` output *into* `Set-`/`Remove-`/`Rename-`/`Add-`.

### Resilient REST API Integration

- [ ] **Audit remaining single-colon `filter=` usages for the exact-vs-fuzzy bug**
  - Four of the five known single-colon filter call sites were fixed (see Completed): `Get-DMFileSystemSnapshot`, `Get-DMLunSnapshot`, and `Get-DMLunbyFilter` now request an exact double-colon match by default. `Get-DMlunByWWN` and the new `Get-DMlunByName` were rebuilt as thin wrappers over `Get-DMLunbyFilter`, so they inherit the same wildcard-aware exact/fuzzy behavior for free.
  - One lower-risk one remains unfixed: `Get-DMAlarm` (`alarmStatus:$statusAlarm` — a status enum, collision risk is low but not zero depending on the exact set of status strings). Hasn't caused an observed problem; fix if/when convenient, same `::`-for-exact pattern as the others.
- [ ] **Audit `Get-DM*` Commands for Missing Pagination**
  - Only `Get-DMlun`, `Get-DMFileSystem`, and `Get-DMLunSnapshot` use `Private/Invoke-DMPagedRequest.ps1`; confirmed via a live 1000+ LUN test that this pagination path had two real bugs (wrong range separator, off-by-one end boundary — both fixed and confirmed against `OceanStor Dorado 6.1.6 REST Interface Reference.md`, which documents `range=[start-end]` with an **exclusive** end index and recommends querying a maximum of 150 records per unpaged request).
  - The other ~35 list-returning `Get-DM*` commands (`Get-DMhost`, `Get-DMnfsFileClient`, `Get-DMShare`, `Get-DMPortGroup`, `Get-DMdisk`, `Get-DMHostInitiator`, etc.) call `Invoke-DeviceManager` directly for a single unpaged request — each was a candidate for the same silent-truncation failure mode (no error, just an incomplete list) that pagination exists to prevent.
  - **Partially tested live, evidence is reassuring but not conclusive**: after bulk-creating real hosts, filesystems, NFS shares, NFS clients, and snapshots on the array and re-running the full mutation-integrity suite, three of the never-tested-past-150 commands came back with unpaged counts well past that threshold and zero errors in the trace (checked all 8,757 `error` blocks in `mutation-trace-last-result.json`, all `code: 0`): `Get-DMhost` returned 213, `Get-DMShare:NFS` returned 217, `Get-DMnfsFileClient` returned 1394. So on this firmware, at least these three unranged endpoints do **not** truncate the way `lun` did — the truncation risk is not universal across every non-paginated `Get-DM*` command as originally feared.
  - **Still open**: this doesn't clear the other ~32 untested commands (`Get-DMPortGroup`, `Get-DMdisk`, `Get-DMHostInitiator`, etc.), and doesn't rule out truncation at some higher threshold above 1394 for the three that were tested — the array's actual per-resource unranged cap (if any) is still unknown. Bulk-create test data for the remaining resource types, and consider pushing one of the three already-confirmed-safe types (e.g. NFS clients) well past 1394 to find where — or whether — a cap exists at all.

- [x] ~~Fix `Get-DMhostbyId` silently returning wrong results (array's ID filter does a substring match, not exact).~~ Found while building `Get-DMhostbyFilter.ps1` as a shared base for `Get-DMhostbyId`/`Get-DMhostbyName`: confirmed live that `host?filter=ID:5` returns every host whose ID *contains* "5" (matched 40 hosts on this array — 5, 15, 51, 59, 150-159, 250-259, etc.) instead of exactly one. Root cause turned out to be documented, not a quirk: per the OceanStor REST API reference, a single colon in `filter=field:value` requests a fuzzy (substring) match, a double colon requests an exact match — confirmed live (`ID::5` returns exactly one host; `NAME:ESX` fuzzy-matches 9 hosts; `NAME::ESX` exact-matches 0, since no host is named exactly that). This bug predates today's refactor — `Get-DMhostbyId.ps1` sent the single-colon query before `Get-DMhostbyFilter` existed. Fixed properly (not just papered over): `Get-DMhostbyFilter` now sends an exact double-colon query when `-Keyword` has no wildcard, and a single-colon fuzzy hint (the literal substring between the wildcards) only when `-Keyword` has a leading/trailing `*`; any other wildcard shape (`?`, a `[...]` class, or a `*` in the middle) falls back to a full fetch since it can't be expressed as one substring. A client-side `-Like` re-check is always applied regardless, so an imprecise server-side result only costs extra candidates, never a wrong final answer. Same fix applied to `Get-DMLunbyFilter` (rebuilt to match), and to two real correctness risks that used the same single-colon pattern on short numeric IDs: `Get-DMFileSystemSnapshot` (`PARENTID`) and `Get-DMLunSnapshot` (`SOURCELUNID`) — both could have silently mixed snapshots from an unrelated filesystem/LUN into a result whenever one Id was a substring of another's. Verified live throughout.
- [x] ~~Fix `ArgumentCompleter` blocks that silently never worked because they called a private (unexported) function.~~ Found while adding tab-completion for `Get-DMhostbyFilter`/`Get-DMLunbyFilter -Filter`: a PowerShell class type literal or a call to an unexported module function, referenced directly inside an `[ArgumentCompleter({...})]` attribute scriptblock, does not resolve when the real completion engine invokes it — confirmed with `[System.Management.Automation.CommandCompletion]::CompleteInput` and isolated with a minimal repro (a completer calling an unexported function silently falls back to default file-path completion; the identical scriptblock calling an *exported* function works). This is invisible to Pester tests that invoke a completer via `InModuleScope`, since that context doesn't have the restriction — a completer can look correct in tests and still never work for a real user pressing Tab. A full audit (`grep` every `ArgumentCompleter` block in `Public/*.ps1` for called function names not matching a `Public/*.ps1` file) found one other pre-existing instance: `Add-DMPortToPortGroup`/`Remove-DMPortFromPortGroup`'s `-PortName` completers called the private `Get-DMPortGroupCandidate`. Fixed both without adding new public commands (the user's explicit preference over exposing internal helpers): `Get-DMhostbyFilter`/`Get-DMLunbyFilter`'s completers now derive property names from one live sample object returned by the already-public `Get-DMhost`/`Get-DMlun`; the two port completers have `Get-DMPortGroupCandidate`'s 3-line body inlined directly (it only calls already-public `Get-DMPortFc`/`Get-DMPortETH`/`Get-DMLif`, so no further extraction was needed). `Get-DMPortGroupCandidate` itself is untouched and still used normally by `ValidateScript` blocks and function bodies elsewhere, which don't have this restriction — only `ArgumentCompleter` scriptblocks specifically are affected. Verified live with the real completion API against the array for all four completers.

### Performance

- [x] ~~Fix host-command validation making an N+1 bulk `Get-DMhost` fetch instead of a targeted lookup.~~ `Get-DMhost` (fetch-all) enriches every returned host with FC/iSCSI initiators via `Set-DMHostInitiator`, which issues 2 REST calls per host — at 213+ hosts on the test array that's 400+ extra round-trips per single `Get-DMhost` call (~30s). 11 commands (`Remove-DMHost`, `Remove-DMNvmeInitiatorFromHost`, `Remove-DMIscsiInitiatorFromHost`, `Remove-DMFiberChannelInitiatorFromHost`, `New-DMIscsiInitiator`, `New-DMFiberChannelInitiator`, `Add-DMHostToHostGroup`, `Remove-DMHostFromHostGroup`, `Get-DMIscsiInitiator`, `Get-DMFiberChannelInitiator`, `Get-DMNvmeInitiator`) called this full fetch 1-3 times per invocation just to resolve one host by name, via a `ValidateScript` block and/or the function body. ## Rejected

- [x] ~~Build Dynamic Parameter-to-Payload Transformer.~~ **Rejected** — audited the actual `$body` construction across `New-DM*`/`Set-DM*` commands (e.g. `New-DMLun.ps1`, `New-DMFileSystem.ps1`, `New-DMHost.ps1`) and found the parameter-to-payload mapping is not mechanical: key renames don't follow PascalCase→UPPERCASE (`-LunName` → `NAME`, `-StoragePoolID` → `PARENTID`, `-EnableCache` → `ENABLE_CACHE`), most fields need value transforms (capacity→blocks, friendly enum strings → Huawei numeric codes, bool→int casts) or are hardcoded constants unrelated to any parameter (`TYPE = 21`), and `Set-*` commands rely on `$PSBoundParameters.ContainsKey(...)` gating per field for PATCH semantics. A generic reflection-driven transformer would only cover the small mechanical slice and still need per-field override logic for everything else — net more indirection, not less. Established REST-wrapping PowerShell modules (Az, PowerCLI, NetApp/Pure SDKs) don't hand-roll this either; they use SDK-generated clients or hand-write the body per command, which is what this module already does.
- [x] ~~Build Active Token Lifecycle Validation.~~ **Rejected** — re-authentication using "securely cached session parameters" directly conflicts with a design decision this module already made: `Private/class-OceanstorSession.ps1` has a commented-out `$Credentials` field with the author's own reasoning attached ("dont want to have the credentials in memory for long time"), matching `ANALYSIS.md` finding S2 (plaintext password wiped immediately after login). Silent re-auth needs a reusable secret retained for the session's lifetime — reopening the exact exposure that decision closed — and Huawei's login flow (Basic-Auth-once-for-an-opaque-token) has no refresh-token primitive to build on the way OAuth2-based CLIs (Azure/AWS/GCP) do. The `Test-DMSession` expiry-check half is also weak on its own: the module never captures a token TTL from the login response (none is returned), so "close to expiry" means either hardcoding an undocumented, unverifiable assumed timeout or proactively pinging before every real call — an inherently racy check (TOCTOU) that duplicates what "attempt the call, handle the failure" already does. See the narrower message-quality fix implemented instead, in Completed below.
