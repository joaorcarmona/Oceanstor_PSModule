## Completed / Validated

### Native Pipeline & Context Architecture

- [x] ~~Fix Multi-Object Pipeline Input (`Get-DMlun | Remove-DMLun` silently drops all but one item).~~ All 46 `Set-`/`Remove-`/`Rename-`/`Add-` commands (every one of that shape except `Set-DMdnsServer`, a singleton config replace with no per-object identity to bind) now wrap their body in `begin`/`process`/`end` instead of a flat body, so every piped object is processed instead of only the last one bound. `WebSession` lost its `ValueFromPipeline` (by-value) binding on all of them â€” that binding was live and actively wrong: a piped typed object (a LUN, a host, ...) was coercing into `-WebSession` by value instead of into the real identifying parameter, which was the actual mechanism behind the silent single-item behavior, not just the missing `process{}` block. The identifying parameter on each command now has `ValueFromPipelineByPropertyName = $true` plus an `[Alias(...)]` matching the *real* class property name (`Name` for most objects, `Id` for the initiator commands, `'Share Path'` for `Remove-DMNfsShare`) â€” verified per command against the actual returned class, not assumed. Per-item session resolution follows explicit `-WebSession` argument > the piped object's own session property > `$script:CurrentOceanstorSession`, so both "connect and forget" and concurrent multi-array session usage work correctly per item.
  - **Error handling**: a failing item does not abort the batch. Each command's `process{}` wraps its mutating logic in try/catch and reports failures via `$PSCmdlet.WriteError()`, matching `Remove-Item -ErrorAction Continue` semantics â€” confirmed live that 2 valid LUNs piped alongside 1 deliberately invalid name still both get removed, with the invalid one reported as a non-terminating error. This is a deliberate behavior change from the old single-item throw-and-stop semantics; existing unit tests asserting `Should -Throw` for what are now per-item pipeline failures were updated to assert a non-terminating error via `-ErrorVariable` instead (an explicit `-ErrorAction Stop` still escalates to a terminating exception for scripts that want that).
  - **ValidateScript network calls moved into `process{}`**: several commands' `ValidateScript` blocks used to re-fetch the candidate list and depended on `$WebSession`'s bound value; once `WebSession` became per-item pipeline-bound, that created an undefined binding-order hazard (confirmed empirically â€” `ValidateScript` for one pipeline-bound parameter can run before or after another pipeline-bound parameter is set for the same object, with no ordering guarantee). Existence/ambiguity resolution for the identifying parameter(s) now happens inside `process{}`, after all binding for that item is guaranteed complete; `ArgumentCompleter` blocks (interactive-only, unaffected by this hazard) were left in place.
  - **Redesigned the 3 commands using an unsafe caching shortcut**: `Add-DMHostToHostGroup`, `Add-DMPortToPortGroup`, `Remove-DMHostFromHostGroup` used to stash `ValidateScript` lookup results in shared `$script:_dm*` module-scope variables to avoid a redundant fetch. That was never safe for concurrent per-item pipeline processing; resolution now happens directly in `process{}` (still exactly one fetch per item, same round-trip count as before).
  - **Multi-identifier commands**: for commands that resolve two named objects (e.g. `Remove-DMLunFromLunGroup`, `Add-DMHostGroupToMappingView`), the object naturally flowing through the pipe (the member being added/removed) is the pipeline-bound primary parameter; the container (the group/mapping view) stays a plain required argument, since it applies to the whole batch in one command invocation.
  - Verified with 515 unit tests (including new multi-item, continue-on-error, and concurrent-session cases per command) and twice against the real array via `Tests/Integration/Invoke-GetterIntegrityValidation.ps1 -RunMutatingTests`: once with a dedicated pipeline-batch section (3 test-owned LUNs piped through `Set-DMLun`, `Add-DMLunToLunGroup`, and `Remove-DMLun` with one deliberately invalid name mixed in) added to `Tests/Integration/Private/Workflows/LunGroup.ps1`, and once for the full mutation suite exercising all 46 refactored commands' normal single-item paths â€” both runs: 0 failures, 0 remaining test-owned resources.

- [x] ~~Fix `Get-DMhostbyId` silently returning wrong results (array's ID filter does a substring match, not exact).~~ Found while building `Get-DMhostbyFilter.ps1` as a shared base for `Get-DMhostbyId`/`Get-DMhostbyName`: confirmed live that `host?filter=ID:5` returns every host whose ID *contains* "5" (matched 40 hosts on this array â€” 5, 15, 51, 59, 150-159, 250-259, etc.) instead of exactly one. Root cause turned out to be documented, not a quirk: per the OceanStor REST API reference, a single colon in `filter=field:value` requests a fuzzy (substring) match, a double colon requests an exact match â€” confirmed live (`ID::5` returns exactly one host; `NAME:ESX` fuzzy-matches 9 hosts; `NAME::ESX` exact-matches 0, since no host is named exactly that). This bug predates today's refactor â€” `Get-DMhostbyId.ps1` sent the single-colon query before `Get-DMhostbyFilter` existed. Fixed properly (not just papered over): `Get-DMhostbyFilter` now sends an exact double-colon query when `-Keyword` has no wildcard, and a single-colon fuzzy hint (the literal substring between the wildcards) only when `-Keyword` has a leading/trailing `*`; any other wildcard shape (`?`, a `[...]` class, or a `*` in the middle) falls back to a full fetch since it can't be expressed as one substring. A client-side `-Like` re-check is always applied regardless, so an imprecise server-side result only costs extra candidates, never a wrong final answer. Same fix applied to `Get-DMLunbyFilter` (rebuilt to match), and to two real correctness risks that used the same single-colon pattern on short numeric IDs: `Get-DMFileSystemSnapshot` (`PARENTID`) and `Get-DMLunSnapshot` (`SOURCELUNID`) â€” both could have silently mixed snapshots from an unrelated filesystem/LUN into a result whenever one Id was a substring of another's. Verified live throughout.
- [x] ~~Fix `ArgumentCompleter` blocks that silently never worked because they called a private (unexported) function.~~ Found while adding tab-completion for `Get-DMhostbyFilter`/`Get-DMLunbyFilter -Filter`: a PowerShell class type literal or a call to an unexported module function, referenced directly inside an `[ArgumentCompleter({...})]` attribute scriptblock, does not resolve when the real completion engine invokes it â€” confirmed with `[System.Management.Automation.CommandCompletion]::CompleteInput` and isolated with a minimal repro (a completer calling an unexported function silently falls back to default file-path completion; the identical scriptblock calling an *exported* function works). This is invisible to Pester tests that invoke a completer via `InModuleScope`, since that context doesn't have the restriction â€” a completer can look correct in tests and still never work for a real user pressing Tab. A full audit (`grep` every `ArgumentCompleter` block in `Public/*.ps1` for called function names not matching a `Public/*.ps1` file) found one other pre-existing instance: `Add-DMPortToPortGroup`/`Remove-DMPortFromPortGroup`'s `-PortName` completers called the private `Get-DMPortGroupCandidate`. Fixed both without adding new public commands (the user's explicit preference over exposing internal helpers): `Get-DMhostbyFilter`/`Get-DMLunbyFilter`'s completers now derive property names from one live sample object returned by the already-public `Get-DMhost`/`Get-DMlun`; the two port completers have `Get-DMPortGroupCandidate`'s 3-line body inlined directly (it only calls already-public `Get-DMPortFc`/`Get-DMPortETH`/`Get-DMLif`, so no further extraction was needed). `Get-DMPortGroupCandidate` itself is untouched and still used normally by `ValidateScript` blocks and function bodies elsewhere, which don't have this restriction â€” only `ArgumentCompleter` scriptblocks specifically are affected. Verified live with the real completion API against the array for all four completers.
